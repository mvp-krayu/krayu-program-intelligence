# Projection vs Expansion Governance

## Projection Blockage (Class A)

Condition: mandatory qualification artifacts absent.
Impact: projection cannot be authorized.
Visual: red escalation, critical posture, "QUALIFICATION INCOMPLETE".
Example: FastAPI at S1 — missing decision_validation, reproducibility_verdict,
semantic_continuity_crosswalk.

## Expansion Constraint (Class C)

Condition: structural grounding incomplete for next S-state.
Impact: current projection authorized, future maturation constrained.
Visual: blue progression, constrained posture, "EXPANSION CONSTRAINED".
Example: BlueEdge at S2 — 13 grounding gaps for S3, but S2-qualified and
projection-authorized.

## Key Distinction

Projection blockage = system cannot be used for executive projection.
Expansion constraint = system is usable but cannot progress to full governance.

These must never share visual severity framing.
