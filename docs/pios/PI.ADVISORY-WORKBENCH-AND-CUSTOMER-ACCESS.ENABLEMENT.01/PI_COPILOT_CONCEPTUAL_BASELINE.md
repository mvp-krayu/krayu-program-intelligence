## Purpose

This document captures the frozen conceptual model of the Program Intelligence Co-Pilot. The model is complete. Future discussions focus on implementation, UX, context assembly, retrieval, publishing, and visualization — not on redefining what the PI Co-Pilot is.

---

## 1. DEFINITION

The **Program Intelligence Co-Pilot** is the universal intelligence interaction surface for Program Intelligence. It can interrogate anything Program Intelligence knows — from a single surface.

The Co-Pilot is:
- An **operator cognition surface** — not a feature, not a tool, not a chatbot
- The **internal publishing layer** for PI — transforming governed intelligence into consumption artifacts
- **Progressively contextual** — always useful, increasingly rich as more knowledge becomes available
- **Architecturally independent** of any single artifact — operates before, during, and after verdict generation

The Co-Pilot is NOT:
- A consultant tool
- An advisory platform
- A sales assistant
- A retrieval assistant
- A verdict-consumption-only tool

Those are secondary effects of a broader capability.

## 2. PROGRESSIVE CONTEXT MODEL

The Co-Pilot operates at 4 context levels. Not all levels require a specimen or verdict.

### Level 0 — No Specimen

**Available knowledge:** Doctrine + Commercial + Runtime architecture + Vault

**Example interactions:**
- "What is Program Intelligence?"
- "What is the difference between SA and SA-DD?"
- "How does PI compare to SonarQube?"
- "Explain a pressure zone."
- "What is the commercial model?"
- "What is the governance model?"
- "How do I explain this category to a PE firm?"
- "Explain CIP. Explain PICR. Explain PICP."
- "Explain convergence."

**The Co-Pilot is useful at Level 0.** No specimen, no verdict, no findings — but PI doctrine, commercial packaging, governance model, and competitive positioning are all queryable.

**Curate at Level 0:**
- "Create a PE explanation of Program Intelligence."
- "Generate a category positioning brief."
- "Compare Program Intelligence to SonarQube for a CTO audience."
- "Explain Structural Intelligence to a CTO."
- "Generate a partner onboarding brief."
- "Draft website copy for the SA offer."

Curate exists at every level. At Level 0 it produces consumption artifacts from doctrine, commercial, and category knowledge. The source material changes per level, not the capability.

### Level 1 — Specimen Loaded (No Verdict)

**Additional knowledge:** + Specimen (topology, signals, findings, conditions, consequences, qualification state)

**Example interactions:**
- "What is emerging in this topology?"
- "Which clusters dominate?"
- "What appears unusual?"
- "What should I investigate?"
- "What changed since last run?"
- "What is the qualification state?"

**The Co-Pilot gains specimen awareness.** Answers become grounded in THIS program's structural reality. Still no verdict — but topology, signals, and qualification state are all queryable.

### Level 2 — Verdict Generated

**Additional knowledge:** + Verdict (9-chapter content, posture, findings, operational ceiling, governance boundary)

**Example interactions:**
- "Explain the verdict."
- "Defend the ESCALATE posture."
- "Generate an Acquisition Risk Summary."
- "Generate a Board Summary."
- "Generate a CTO Summary."
- "Generate an Investment Committee Brief."
- "What would the IC challenge?"

**The Co-Pilot reaches its richest context.** Curate becomes the center of gravity — producing consumption artifacts from the governed verdict. This is the primary post-verdict value: same truth, different consumption.

### Level 3 — Consumption Artifacts Exist

**Additional knowledge:** + Publishing assets (previously generated consumption artifacts, curated narratives, visual assets)

**Example interactions:**
- "Enhance the IC Brief with topology graphics."
- "Package the Acquisition Risk Summary as a one-pager."
- "Compare the Board Summary to the CTO Summary."
- "What has been curated so far?"

**The Co-Pilot operates on its own output.** The Publish tier becomes fully active — enhancing, formatting, and refining consumption artifacts.

### Progressive Enrichment Principle

The Co-Pilot is always useful. Its answers become richer as more context becomes available. The continuity — same surface from "What is PI?" through "Generate a Board Pack" — IS the product.

```
Level 0: Doctrine + Commercial + Vault + Runtime
Level 1: + Specimen
Level 2: + Verdict
Level 3: + Publishing assets
```

## 3. INTERACTION HIERARCHY

The Co-Pilot has 9 interaction modes across 3 tiers. Not all modes require all context levels.

```
PI KNOWLEDGE GRAPH
        ↓
PI CO-PILOT

1. UNDERSTAND
   ├── Query     — Ask anything across the knowledge graph
   ├── Explore   — Navigate and discover
   └── Explain   — Understand concepts, findings, architecture

2. CURATE (center of gravity when verdict exists)
   ├── Compare   — Differentiate and contrast
   ├── Curate    — Produce audience-specific consumption artifacts
   └── Challenge — Stress-test findings, handle objections

3. PUBLISH (extends Curate)
   ├── Visualize — Enhance with structural graphics
   ├── Package   — Format into deliverable artifacts
   └── Position  — Frame commercial implications
```

### Per-Mode Detail

| Tier | Mode | Min Level | Definition |
|------|------|----------|-----------|
| **Understand** | **Query** | 0 | Ask anything the knowledge graph can answer |
| **Understand** | **Explore** | 0 | Navigate the knowledge graph — discover connections, trace lineage |
| **Understand** | **Explain** | 0 | Understand concepts, findings, or architecture in depth |
| **Curate** | **Compare** | 0 | Differentiate and contrast (PI vs competitors, SA vs SA-DD, cluster vs cluster) |
| **Curate** | **Curate** | 0 | Produce audience-specific consumption artifacts from available knowledge |
| **Curate** | **Challenge** | 1 | Stress-test findings and handle objections with evidence |
| **Publish** | **Visualize** | 2 | Enhance consumption artifacts with structural graphics |
| **Publish** | **Package** | 2 | Format curated content into deliverable artifacts |
| **Publish** | **Position** | 0 | Frame commercial implications |

### Tier Relationships

- **Understand** supports Curate — the operator must understand before curating
- **Curate** exists at every context level — what changes is the source material (doctrine/commercial at L0, specimen at L1, verdict at L2, artifacts at L3). Curate becomes the center of gravity when a verdict exists, but it does not require one.
- **Publish** extends Curate — it enhances and formats what Curate produces
- **All tiers** are available at Level 0 (except Visualize and Package which require Level 2 — they depend on structural graphics and verdict content)

### Consumption Artifact Examples

The term "consumption artifact" replaces "audience-specific narrative." The artifact catalog is **open** — these are examples, not a fixed product inventory:

- Acquisition Risk Summary
- Investment Committee Brief
- Board Summary
- CTO Summary
- Executive Summary
- Operational Risk Summary
- Transformation Summary
- Briefing note
- Board pack
- Acquisition memo
- Executive one-pager
- Workshop deck
- Slide deck
- Graphic narrative
- Website copy
- Proposal insert
- RFP response
- Investment deck
- Conference talk
- Category narrative
- Partner onboarding brief
- Infographic

The real capability is: take anything Program Intelligence knows → transform it into whatever consumption artifact is needed. Same knowledge. Different consumption. The catalog grows with operational use.

### Curate at Every Level

Curate is not verdict-dependent. What changes per level is the source material, not the capability:

| Level | Source Material | Example Curate Output |
|-------|---------------|----------------------|
| 0 | Doctrine, commercial, category | PE explanation, category positioning brief, competitive comparison, partner onboarding brief |
| 1 | + Specimen | Specimen topology narrative, investigation summary, structural risk overview |
| 2 | + Verdict | IC Brief, Board Summary, CTO Summary, acquisition risk assessment |
| 3 | + Publishing assets | Refined artifacts, enhanced with prior output, cross-artifact comparison |

## 5. OPERATOR COGNITION SURFACE

The PI Co-Pilot is an operator cognition surface — not a feature. This distinction determines how it is built.

**Feature thinking:** Build small, ship fast, iterate. Optimize for time-to-market.

**Operator cognition surface thinking:** Understand the role it plays in the ecosystem first. The progressive context continuity — same surface from "What is PI?" through "Generate a Board Pack" — IS the product. Build to serve that continuity.

### What Makes It Different from "Claude with Docs"

A generic AI assistant has no context. The operator must manually establish: what client, what run, what posture, what findings, what SKU, what audience, what governance boundaries.

The PI Co-Pilot pre-loads operational context from the PI knowledge graph BEFORE any question is asked:

- **At Level 0:** PI doctrine, commercial model, governance architecture, competitive positioning
- **At Level 1:** + THIS specimen's topology, signals, findings, qualification state
- **At Level 2:** + THIS verdict's posture, chapters, operational ceiling, governance boundary
- **At Level 3:** + previously generated consumption artifacts

Every response inherits this assembled context. That is the moat.

### Assembled Context = The Moat

At Level 2 (richest context), the Co-Pilot knows:

| Dimension | What It Carries |
|-----------|----------------|
| **Current Verdict** | 9-chapter content, posture, operational ceiling |
| **Current Specimen** | Client identity, domain count, topology |
| **Current Run** | Run ID, timestamp, evidence coverage, qualification |
| **Current Topology** | Domains, clusters, edges, pressure zones |
| **Current Findings** | Named conditions, severity, location, consequence attribution |
| **Current Offer** | SKU, pricing, included surfaces, access terms |
| **Target Audience** | CTO, VP Eng, PE partner, deal team, IC, board |
| **Interaction Mode** | Query, explore, explain, compare, curate, challenge, visualize, package, position |
| **Commercial Authority** | Offers, objection handling, competitive positioning |
| **PI Doctrine** | 13 prohibitions, governance boundaries, authority model |

## 5B. TRANSFORMATION MODEL

The PI Co-Pilot is not a query tool. It is a **transformation layer** over the PI knowledge graph.

### The Transformation Primitive

Every Co-Pilot interaction is a transformation request. The operator provides:

| Input | What It Specifies |
|-------|------------------|
| **Intent** | What transformation is needed (explain, compare, curate, challenge, position, visualize, package) |
| **Audience** | Who will consume the output (CTO, PE partner, IC, board, engineer, partner, internal) |
| **Context** | What knowledge level is available (L0 doctrine, L1 specimen, L2 verdict, L3 publishing assets) |

The Co-Pilot transforms available PI knowledge into the required artifact. A query is simply one transformation request.

### Operator Amplification

The destination is not transformation. Transformation is the mechanism. The destination is **Operator Amplification**.

```
OPERATOR
    (intent)
        +
PI CO-PILOT
    (knowledge + context + transformation)
        =
AMPLIFIED OPERATOR
    (any artifact, any audience, any format)
```

The operator enters with intent. The Co-Pilot contributes knowledge, context, and transformation capability. The outcome is an operator who can produce — from a single surface — narrative generation, positioning, comparison, onboarding, commercial packaging, proposals, presentations, decks, graphics, workshops, and category authority. Without redesigning the architecture for each new capability.

The transformation catalog is **open**. The artifact catalog is **open**. Both grow with operational use. Attempting to enumerate all possible transformations or artifacts would artificially constrain the architecture.

### What Is Fixed vs What Is Open

| Fixed (Constitutional) | Open (Evolving) |
|----------------------|----------------|
| The 7 knowledge graph domains | The specific transformations possible |
| The 4 context levels | The specific artifacts producible |
| The 13 absolute prohibitions | The audience vocabulary |
| The governance boundaries | The format vocabulary |
| "Never creates intelligence" | The consumption patterns |
| Evidence traceability | The delivery mechanisms |

The transformation layer becomes more capable as the knowledge graph grows — without changing its architecture.

### Transformation, Not Questions

The 9 interaction modes (Query, Explore, Explain, Compare, Curate, Challenge, Visualize, Package, Position) are **transformation types**, not question categories. The distinction matters:

- **Question thinking:** "What can the operator ask?" → leads to fixed query catalogs
- **Transformation thinking:** "What can the operator transform PI knowledge into?" → leads to an open architecture

The PI Co-Pilot is the universal transformation layer sitting on top of the PI knowledge graph. Every mode is a different class of transformation. The operator specifies intent + audience + context. The Co-Pilot produces the artifact. The purpose is not transformation for its own sake — it is operator amplification. The operator becomes more capable because the Co-Pilot carries the knowledge, the context, and the transformation machinery.

## 6. GOVERNANCE BOUNDARIES

### Constitutional Invariant

The Co-Pilot never creates intelligence. It never changes findings. It never changes evidence. It never changes qualification. It interrogates, explains, curates, and packages what already exists in the PI knowledge graph.

### The Strategic Test

What value can the Co-Pilot create without changing a single finding, a single piece of evidence, or a single qualification state?

- Better understanding, explanation, comparison → **Co-Pilot**
- Audience-adapted consumption artifacts → **Co-Pilot**
- Visual enhancement, formatted deliverables → **Co-Pilot**
- Challenge handling, commercial positioning → **Co-Pilot**
- New findings, new evidence, new qualification, new verdicts → **Runtime (not Co-Pilot)**

### Operational Rules

| Rule | Enforcement |
|------|-------------|
| Never creates intelligence | Interrogates the knowledge graph — never generates findings, evidence, or qualification |
| Never changes findings | Consumption artifacts must be traceable to verdict findings |
| Never changes evidence | Evidence chain is immutable — Explain and Explore surface it, never modify it |
| Never changes qualification | SQO state is consumed, not produced |
| Never visible to customer | Co-Pilot output is operator-only |
| Evidence-compatible | All outputs must be compatible with structural evidence |
| Audience-adapted, not truth-adapted | Changes consumption format, never changes what is said |
| Context-honest | Discloses what context level it operates at — never hallucinates context it does not have |
| 13 absolute prohibitions | Same as LENS 75.x — no team behavior, org intent, human motive, cultural diagnosis, personnel attribution |

## 7. NON-GOALS

The PI Co-Pilot is NOT:

| Non-Goal | Why |
|----------|-----|
| A customer-facing feature | Operator-only. Customers consume LENS and Assessment Package. |
| A consulting platform | Produces consumption artifacts, not consulting engagements |
| A qualification engine | Consumes SQO state, never produces it |
| A pipeline operator | Does not trigger runs, modify evidence, or alter findings |
| A replacement for LENS | LENS is the customer surface. The Co-Pilot is the operator surface. Distinct roles. |
| A replacement for the Assessment Package | The Structural Verdict and Evidence Record are governed artifacts. The Co-Pilot produces derivative consumption artifacts, not replacements. |
| A generic chatbot with PI docs | The assembled operational context (progressive, pre-loaded, knowledge-graph-backed) is the moat — not retrieval |
