## 4. Relationship To Software Intelligence

Software Intelligence and Runtime Cognition are **complementary, not competing.**

| Dimension | Software Intelligence | Runtime Cognition |
|---|---|---|
| Evidence source | Static import graph + structural enrichment | Runtime coordination topology |
| Question answered | "What pressures exist inside the software structure?" | "What pressures exist outside the organization's awareness?" |
| Conditions produced | COUPLING, PROPAGATION, FRAGILITY, CONVERGENCE, DRIFT | RUNTIME_DEPENDENCY, RUNTIME_COORDINATION, RUNTIME_CONCENTRATION, RUNTIME_BOUNDARY_EXTENSION |
| Blindness scope | N/A — SW-Intel sees what's visible | BOUNDARY, SILENCE, COUPLING, GRAVITY_DIVERGENCE |
| Activation | Structural enrichment available | Runtime evidence + divergence + blindness |
| When active together | SW-Intel provides "what you can see" | Runtime Cognition provides "what you cannot see" |

The EIR narrative architecture already codifies this: in EXECUTION_BLINDNESS mode, SW-Intel is Chapter 5 ("the static picture is not wrong — it is incomplete"). SW-Intel establishes the competent baseline. Runtime Cognition reveals what lies beyond it.

**Absorbing Runtime Cognition into Software Intelligence would be constitutionally wrong because:**
1. SW-Intel is an import-graph-derived cognition domain. Runtime Cognition is not.
2. SW-Intel produces conditions about code structure. Runtime Cognition produces conditions about operational coordination.
3. SW-Intel is always available when structural enrichment exists. Runtime Cognition only activates when runtime evidence produces blindness findings.
4. SW-Intel does not produce blindness classes. Runtime Cognition does.
5. Commercially, SW-Intel is the visible structural intelligence. Runtime Cognition is the invisible operational intelligence. Merging them loses the commercial distinction that creates pricing power.

## 11. Classification Options

### Option A: Capability

Runtime Cognition is just a feature — a set of functions that enrich the existing assessment.

**Against:** A capability does not have its own activation contract, its own blindness taxonomy, its own EIR narrative mode, and its own cognition primitives. Runtime Cognition has all four. It is too structured and too independent to be "just a capability."

### Option B: Cognition Package

Runtime Cognition is a packaged set of cognition objects that can be loaded when evidence supports it.

**Partial fit:** This captures the evidence-driven activation and the cognition object production. But it understates the consumer impact — Runtime Cognition doesn't just produce objects; it changes how THORR, LENS, and EIR operate. A cognition package typically feeds consumers without changing their mode. Runtime Cognition activates a different narrative mode.

### Option C: First-Class PI Module

Runtime Cognition is a Domain Cognition Module at the same architectural level as Software Intelligence, with its own evidence domain, cognition primitives, blindness taxonomy, and consumer projection impact.

**Strong fit.** It passes all seven tests (independence, necessity, activation, persistence, projection, marketplace, boundary). It has the same structural relationship to PI Core that Software Intelligence does — it produces domain-specific cognition that the core substrate processes and consumers project.

### Option D: Specialization of Software Intelligence

Runtime Cognition is absorbed into Software Intelligence as a runtime-aware extension.

**Against:** SW-Intel is import-derived. Runtime Cognition is runtime-derived. They answer different questions from different evidence through different primitives producing different conclusions. Merging them conflates "visible structural pressures" with "invisible operational blindness" — destroying the commercial distinction that makes Execution Blindness valuable.

## 12. Final Recommendation

**Classification: C — Runtime Cognition is a first-class Program Intelligence Domain Cognition Module.**

Constitutional reasoning:

1. **It passes independence.** No other module produces blindness classifications, gravity divergence findings, or boundary extension detection.

2. **It is necessary.** Without it, PI cannot explain operational reality — only code structure.

3. **It has its own activation.** The locked contract (SYSTEM_CONNECTIVITY + divergence + blindness class) is independent of SW-Intel activation.

4. **It creates persistent cognition.** AF-001..005, qualified domain registry, VLC scope, narrative mode determination.

5. **It changes all consumers.** THORR gains operational gravity. LENS gains blindness/divergence visuals. EIR gains a different narrative spine.

6. **It survives cross-specimen validation.** 4 proven primitives, 0 specimen-specific, correct exclusion on call-graph-dominant systems.

7. **Its boundary is clear.** Runtime evidence domain → runtime cognition primitives → blindness taxonomy → EIR mode activation. Distinct from SW-Intel, distinct from PI Core.
