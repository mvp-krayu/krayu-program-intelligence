# AI-ASSISTED ONBOARDING OPERATING MODEL

> **Status:** LOCKED — Operating model for first 10–50 client onboardings under governed AI assistance.

---

## 1. Purpose

This document defines the transitional operating model for early Program Intelligence client onboardings (first 10–50 clients). During this phase, onboarding is **AI-assisted under governance** — not fully autonomous.

The objective: scale onboarding throughput while preserving governed evidence-first discipline.

## 3. What AI May Do

### 3.1 INSPECT

AI may examine client evidence, source structure, pipeline output, and structural artifacts to identify patterns, anomalies, and gaps.

**Examples:**
- Analyze source tree for unexpected structure (monorepo, generated code, vendor dependencies)
- Examine 40.x topology for structural anomalies
- Review CEU grounding for coverage gaps
- Assess proposition confidence distribution

### 3.2 PROPOSE

AI may generate suggestions, candidates, and draft artifacts that require operator approval.

**Examples:**
- Propose Hero Moment candidates with evidence refs
- Propose learning events from observed friction
- Draft chronicle narrative sections from governed objects
- Suggest enrichment strategies for weak propositions
- Recommend investigation paths for structural anomalies

### 3.3 EXPLAIN

AI may synthesize governed evidence into human-readable explanation under 75.x bounded authority.

**Examples:**
- Explain structural topology in executive terms
- Interpret Hero Moment significance
- Summarize governance findings
- Describe what the pipeline discovered and why it matters

### 3.4 RECONCILE

AI may assist with evidence reconciliation under deterministic contracts.

**Examples:**
- Map SDC candidate CSR to canonical registry
- Cross-reference proposition evidence against structural artifacts
- Identify domain ID mismatches between sources
- Flag confidence inconsistencies

### 3.5 IMPROVE

AI may identify pipeline and process improvements from onboarding experience.

**Examples:**
- Identify missing adapters for new project types
- Surface extraction gaps for specific language/framework patterns
- Propose pipeline parameter adjustments based on observed outcomes
- Flag false-positive structural classifications

## 4. What AI Must NOT Do

| Forbidden Action | Reason |
|-----------------|--------|
| Silently mutate client truth | Evidence-first: all mutations must be traced |
| Bypass governance approval | Non-automatable boundaries: operator reviews all mutations |
| Invent semantic authority | L3 ceiling: AI proposes, never self-authorizes |
| Promote qualification states | S-state transitions require operator governance |
| Overwrite replay lineage | Replay-safe: lineage is append-only |
| Hide assumptions | All AI reasoning must be logged in assistance events |
| Replace operator approval | Non-automatable boundary: 7 boundaries hold |
| Create marketplace/module behavior | Premature: modules emerge from repeated patterns, not AI design |
| Maintain hidden memory | No untracked reasoning influence between sessions |
| Self-escalate authority | AI authority ceiling is fixed per contract, not negotiable |

## 6. Scaling Model (10 → 50 → N)

### Phase 1: First 10 Clients (Current → Near-Term)

- **Model:** Advisory-heavy. Claude Code or API assists each onboarding. Operator present throughout.
- **AI role:** Inspect, explain, propose. All suggestions require explicit approval.
- **Learning focus:** Capture every friction point, every gap, every surprise. Build learning corpus.
- **Chronicle:** Manual or semi-automated from governed artifacts.
- **Target:** Prove genesis replay capture works. Identify repeating patterns.

### Phase 2: Clients 11–30 (Near-Term → Mid-Term)

- **Model:** Pattern-assisted. Repeated onboarding patterns become consumable learnings.
- **AI role:** Propose from learned patterns. Reduced inspection time. Batch suggestions.
- **Learning focus:** Promote REVIEWED learnings to CONSUMABLE. First capability candidates emerge.
- **Chronicle:** AI-assisted narrative drafts from governed objects. Operator review.
- **Target:** Reduce per-client onboarding time. Validate learning promotion pipeline.

### Phase 3: Clients 31–50 (Mid-Term)

- **Model:** Capability-assisted. Reusable capabilities reduce manual intervention.
- **AI role:** Execute proven patterns under governance. Flag novel situations for operator attention.
- **Learning focus:** First MODULE CANDIDATES from repeated capabilities. Marketplace thinking begins.
- **Chronicle:** Templated from proven patterns + novel sections for unique findings.
- **Target:** Validate that repeated patterns produce reliable governed outcomes. Identify marketplace module opportunities.

### Phase 4: Beyond 50 (Future)

- **Model:** Governed automation where proven. Operator attention focused on novel situations.
- **AI role:** Autonomous execution of proven capabilities under replay-safe contracts. Escalation for novelty.
- **Learning focus:** Capability → Module promotion. Marketplace modules operational.
- **Chronicle:** Automated for standard onboardings. Rich for novel specimens.
- **Target:** Scalable governed onboarding with marketplace extension.

