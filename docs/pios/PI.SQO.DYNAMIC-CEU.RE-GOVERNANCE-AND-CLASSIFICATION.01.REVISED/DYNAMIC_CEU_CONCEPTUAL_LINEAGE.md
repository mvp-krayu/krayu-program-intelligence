## 2. Lineage Phases

### Phase 1 — Signal Enrichment Thinking (Early Era)

**Origin:** PI.PSEE-PIOS.SEMANTIC-GOVERNANCE-EXPLORATION.CLOSURE.01
**Authority:** SEMANTIC_GOVERNANCE_EXPLORATION_CLOSURE.md

The earliest antecedent of Dynamic CEU was signal enrichment — the idea that
PSIG signals in the 75.x activation layer could be enriched with semantic
context derived from structural analysis. This was NOT Dynamic CEU. It was
a signal-processing-oriented enrichment model.

**Key characteristics:**
- Enrichment target: PSIG signals, not qualification state
- Enrichment source: structural analysis, not external evidence
- Governance: undefined (exploring whether governance was even needed)
- Replay safety: not yet a concern
- Authority model: none (exploration phase)

**Conceptual contribution to Dynamic CEU:**
- Established that enrichment exists as a category of operation
- Established that enrichment requires governance (via the decision to block it)
- Established the structural-relational intelligence vs semantic authority distinction

### Phase 2 — CEU Grounding Design (Early Era)

**Origin:** PI.CEU.GROUNDING.01
**Authority:** GROUNDING_DESIGN_SPECIFICATION.md

CEU grounding introduced the classification of semantic mappings as
GROUNDED/NON_GROUNDED/DERIVED/INVALID. This was Static CEU — the
deterministic pipeline-certified grounding of structural surfaces to
business semantics.

**Key characteristics:**
- Classification: GROUNDED / NON_GROUNDED / DERIVED / INVALID
- Allowed sources: 6 (MS-1 through MS-6)
- Forbidden sources: 9 (FM-1 through FM-9)
- Pipeline position: between 40.x and 75.x
- Contamination model: strict isolation
- Deterministic: 11-step mapping algorithm

**Conceptual contribution to Dynamic CEU:**
- Established the GROUNDED/NON_GROUNDED classification that Dynamic CEU extends
- Established the forbidden mapping source model that Dynamic CEU must respect
- Created the baseline that Dynamic CEU augments (not replaces)
- Introduced the concept of contamination — which Dynamic CEU solves via overlay isolation

### Phase 3 — Semantic Governance Exploration Closure (Early Era)

**Origin:** PI.PSEE-PIOS.SEMANTIC-GOVERNANCE-EXPLORATION.CLOSURE.01
**Authority:** SEMANTIC_GOVERNANCE_EXPLORATION_CLOSURE.md

The 21-stream exploration family concluded with an intentional block on
semantic activation authority. This is the critical governance inflection
point in the Dynamic CEU lineage.

**Key characteristics:**
- 21 streams explored semantic activation from multiple angles
- 8 phases: boundary discovery → lane clarification → namespace stabilization → enrichment governance → semantic participation governance → replayability → replay determinism → deterministic replay instrumentation
- Final verdict: activation authority BLOCKED
- Forward direction: Deterministic Relational Enrichment

**Conceptual contribution to Dynamic CEU:**
- Established that semantic ACTIVATION is prohibited — Dynamic CEU does not activate semantics
- Established that OBSERVATION and ENRICHMENT are allowed under governance
- Created the governance requirement that Dynamic CEU must satisfy
- Defined the boundary between structural-relational intelligence (allowed) and semantic authority (prohibited)

### Phase 4 — CEU Paradox and Governed Maturation (Mid Era)

**Origin:** PI.SEMANTIC-QUALIFICATION-LOOP.ROADMAP-AND-ARCHITECTURE.01
**Authority:** DYNAMIC_CEU_GOVERNANCE_MODEL.md

The CEU Paradox — high structural CEU grounding does NOT imply semantic
maturity — triggered the reconceptualization of CEU as "Governed Maturation
Assistance." This is the birth of Dynamic CEU as a distinct concept.

**Key characteristics:**
- CEU Paradox: FastAPI 0.90 structural grounding, 0/9 semantic domains grounded
- Reframing: CEU = Governed Maturation Assistance
- 8 allowable behaviors
- 12 prohibited behaviors
- Evidence requirements: input/output/provenance
- 6-step enrichment workflow
- Static/Dynamic CEU relationship model introduced

**Conceptual contribution to Dynamic CEU:**
- Named and defined Dynamic CEU as a concept
- Separated it from Static CEU
- Established the evidence requirement model
- Defined the allowable/prohibited behavior taxonomy
- Introduced the governance-first enrichment workflow

### Phase 5 — Lane Architecture and Replay Safety (Mid Era)

**Origin:** PI.SEMANTIC-QUALIFICATION-LOOP.ROADMAP-AND-ARCHITECTURE.01
**Authority:** SQO_LANE_ARCHITECTURE.md, REPLAY_SAFE_ENRICHMENT_MODEL.md

Lane architecture formalized the structural boundaries that Dynamic CEU
must respect. Replay-safe enrichment guarantees (R1–R5) formalized the
immutability contract that Dynamic CEU must satisfy.

**Key characteristics:**
- Lane A (structural, frozen), Lane B (reference), Lane C (experimental), Lane D (DPSIG, sovereign)
- SQO boundary contract: MAY enrich/annotate/qualify, MUST NEVER mutate/invent/override
- R1: Deterministic derivation
- R2: Additive-only persistence
- R3: Source immutability
- R4: Hash-anchored provenance
- R5: Operation versioning
- Overlay rules OV-01 through OV-06

**Conceptual contribution to Dynamic CEU:**
- Defined the lane boundaries Dynamic CEU operates within
- Established replay safety as a non-negotiable property
- Created the overlay governance model that SEPs formalize

### Phase 6 — Governed Intelligence Extension Lifecycle (Mid Era)

**Origin:** docs/governance/GOVERNED_INTELLIGENCE_EXTENSION_MODEL.md
**Authority:** GOVERNED_INTELLIGENCE_EXTENSION_MODEL.md

The 6-stage lifecycle formalized the process by which any intelligence
extension (including Dynamic CEU) must pass through governance gates.

**Key characteristics:**
- 6 stages: governed manifest → additive lane → replay-safe integration → readiness gating → E2E certification → executive cognitive stabilization
- Lane isolation rules
- TAXONOMY-01 field immutability
- Allowed vs forbidden extension behaviors

**Conceptual contribution to Dynamic CEU:**
- Established the governance lifecycle that Dynamic CEU follows
- Formalized lane isolation as a lifecycle requirement
- Required E2E certification before executive-layer exposure

### Phase 7 — SQO Qualification Governance (Current Era)

**Origin:** PI.SQO.STATE-DETECTION-ENGINE.01, PI.SQO.SEMANTIC-DEBT-ENGINE.01, PI.SQO.MATURITY-SCORING-ENGINE.01
**Authority:** Multiple engine specification artifacts

S-state model, Q-class resolution, semantic debt taxonomy, and maturity
scoring created the qualification framework that Dynamic CEU augments.

**Key characteristics:**
- S-state tiers: S0/S1/S2/S3
- Q-class: pure function of (backed_count, total_count, semantic_continuity_status, evidence_availability)
- Semantic debt: blocking classification, remediation pathways
- Maturity: dimension-level scoring, progression readiness

**Conceptual contribution to Dynamic CEU:**
- Created the evaluation framework that Dynamic CEU overlays contribute to
- Defined the qualification state that re-evaluation computes
- Established the debt inventory that Dynamic CEU can help resolve

### Phase 8 — Semantic Evidence Package Formalization (Current Era)

**Origin:** PI.SQO.SEMANTIC-EVIDENCE-PACKAGE-SPECIFICATION.01
**Authority:** 10 specification documents

SEP architecture is the canonical formalization of Dynamic CEU. It defines
the governed, versioned, provenance-bound container architecture, overlay
mechanics, activation boundaries, and re-evaluation process.

**Key characteristics:**
- Semantic Evidence Package (SEP) architecture
- Static vs Dynamic CEU formal separation
- 7 semantic class authorization model
- Replay-safe overlay architecture
- 5 activation boundaries
- Qualification re-evaluation (4 triggers, 8-step process)
- Multi-package cohabitation rules
- Overlay versioning and rollback

**Conceptual contribution:**
- This IS canonical Dynamic CEU — no further conceptual evolution needed
- All prior concepts either converge here, are superseded, or are classified as prohibited
