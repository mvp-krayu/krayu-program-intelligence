## 1. The Misidentification

The prior stream (PI.EXECUTIVE-INTELLIGENCE-REPORT-COMPILER.01) made a category error. It identified the correct transformation pipeline but named the output incorrectly.

The specification states:

> "The EIC is NOT a document generator. It is a cognition-to-intelligence transformer."

This was the right instinct. But it immediately collapsed back into document generation:

> "...producing multi-format executive intelligence **deliverables**."

The deliverable is not the product. The cognition is the product. The deliverable is a rendered surface of the cognition.

This document performs the forensic separation.

## 4. The Architectural Consequence

The current pipeline model:

```
L0: Structure → L1: Signals → L2: Conditions → L3: Consequences → L3.5: Persona Projection → L4: EIC → Report
```

misses a layer. The corrected model:

```
L0: Structure
 ↓
L1: Signals + Enrichment
 ↓
L2: Conditions
 ↓
L3: Consequences
 ↓
L4: EXECUTIVE COGNITION  ← the 57 objects identified above
 ↓
L5: PROJECTION RENDERING ← the 22 rendering acts
 ↓
Surface: Report / Slides / Memo / Assessment / Workshop
```

**L4 is the missing layer.** It is the canonical runtime object that exists before any rendering decision is made. It contains everything the system KNOWS about a specimen at executive altitude. It is persona-independent, format-independent, and audience-independent.

L5 is the rendering layer. It takes the L4 cognition package and projects it into a specific format for a specific audience. Multiple projections from the same L4 package produce multiple deliverables — all traceable to the same cognition.

## 5. What forBoardroom() Almost Is

The existing `ConsequenceCompiler.forBoardroom()` output is remarkably close to an L4 Executive Cognition object:

| forBoardroom() field | L4 equivalent | Gap |
|---|---|---|
| posture_label | Structural Posture | Close — but posture_label is a rendered string, not a structured object |
| posture_severity | Tension Map (peak severity) | Close — but lacks convergence center detail |
| cognition_slices[] | Finding Inventory | Close — but slices are per-condition, not cross-condition |
| consequence_themes[] | Risk Characterization | Close — but themes are consequence-typed, not risk-stratified |
| domain_narratives[] | Domain-level cognition | Close — but narratives are rendered strings |
| executive_synthesis | Operational Ceiling | Far — synthesis is a rendered paragraph, not a structured assessment |
| combined_synthesis | Tension Summary | Far — synthesis is prose, not structured |

**forBoardroom() is an L3.5 persona projection — it mixes cognition with rendering.** The cognition objects are embedded inside rendered strings. The L4 layer extracts the cognition objects from the rendering.

The relationship:

```
L3: consequenceResult
 ↓
L4: ExecutiveCognitionPackage  ← structured cognition objects
 ↓
L3.5 (revised as L5): forBoardroom() ← rendering of L4 for BOARDROOM persona
                       forBalanced()  ← rendering of L4 for BALANCED persona
                       forReport()    ← rendering of L4 for executive report
                       forSlides()    ← rendering of L4 for presentation
```

forBoardroom() and forBalanced() currently derive directly from consequenceResult. In the corrected architecture, they derive from the ECP. This is not a rewrite — it is an extraction. The cognition objects are already computed inside these functions; they just aren't surfaced as first-class objects.
