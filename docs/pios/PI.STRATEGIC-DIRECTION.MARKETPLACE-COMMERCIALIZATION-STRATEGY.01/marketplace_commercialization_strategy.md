#### PP-1: Watermelon Reporting (RAG Manipulation) — DIRECT KILL SHOT

**The problem:** "Green on the outside, red in the middle." RAG status is subjective. PMs shade toward optimistic because culture treats Red as punishment. One project reports Red while another in identical state reports Green. Permanent green is itself a red flag. Optimism bias compounds until it's too late to intervene.

**Why it's structural:** RAG is opinion. No amount of process standardization fixes this because the underlying data model is subjective human assessment, not structural derivation from execution reality.

**Signäl/PMO solution:** Replace subjective RAG with **Structural Fidelity** + **Q-class** + **S-state**. The system deterministically derives qualification posture from structural evidence. A PMO director doesn't ask "what color is this program?" — they see "this program is at S1 qualification with Q-02 grounding and 4 structural debt items in 2 propagation-critical domains." Not gameable. Not opinion. Structurally derived.

**Build status:** SUBSTANTIALLY BUILT. SQO S-state + Q-class operational. Needs PMO-facing projection.

**Roadmap position:** Tier 1 — Structural Governance Alignment Package, Output E (Qualification Posture).

**The problem:** Most PMO tools treat projects as independent entities. Cross-domain dependencies are tracked in static risk registers (if at all). **Signäl/PMO solution:** Propagation intelligence is the single most differentiated capability. ```

## 8. Evidence Trust & Jira Data Quality — The "Garbage In" Problem

### The core risk

The current evidence model has a **hidden quality gate**: the advisory team. They manually extract, curate, validate, and structure evidence before it enters `source_intake.py`. This human gate ensures that evidence artifacts are structurally meaningful before the system ever sees them.

A Jira connector removes that human gate. The question is: what replaces it?

If nothing replaces it, the system will:
1. Ingest messy Jira data (stale tickets, orphaned epics, manipulated statuses, inconsistent naming)
2. Derive structural intelligence from garbage
3. Present that intelligence as if it were authoritative
4. Get blamed when the intelligence doesn't match reality

**This is the single biggest commercial risk in the integration story.** LENS gets blamed for Jira's sins.

### Why Jira data is structurally untrustworthy by default

| Problem | Description | Structural impact |
|---------|-------------|-------------------|
| **Ontology mismatch** | Jira models *work* (Epic → Story → Task). PiOS models *structure* (Domain → Capability → Component). An Epic is not a Domain. A Story is not a Capability. | Mapping is lossy and assumption-heavy. |
| **Status manipulation** | Teams game status fields. "In Progress" for 6 months. "Done" when it's not. Sprint velocity inflated. | Temporal evidence is unreliable. |
| **Orphaned artifacts** | Stale tickets, abandoned epics, duplicates, unlinked issues. | Topology reconstruction includes noise. |
| **Naming incoherence** | No naming convention. Same capability described 5 different ways. Components named after teams, not functions. | Domain mapping fails or produces false groupings. |
| **Structural flatness** | Many Jira instances have weak hierarchy (everything is a Story, nothing is linked, no Components used). | Insufficient structure to derive topology. |
| **Political data** | Statuses reflect what managers want to see, not execution reality. | Evidence is performative, not structural. |

### The existing governance already provides the framework

The system has the tools to handle this — they just need to be applied to Jira data specifically:

**Q-class classification** (`Q02_GOVERNANCE_AMENDMENT.md` — LOCKED):
- Q-01: Fully grounded — all domains structurally grounded, semantic continuity validated
- Q-02: Partial grounding — some domains grounded, gaps explicitly disclosed
- Q-03: Semantic only — zero structural grounding, semantic continuity present only
- Q-04: Unavailable — evidence absent or rejected

**Rule:** Jira-derived evidence will almost certainly land at Q-02 or Q-03. Rarely Q-01. Sometimes Q-04. The system MUST classify honestly and disclose the classification in every LENS output.

**Evidence source classification** (`evidence_sources.yaml`):
- Current allowed class: `EXPLICIT_OPERATOR_PROVIDED_UPSTREAM_EVIDENCE`
- Jira API data is NOT operator-provided upstream evidence. It's a new source class.
- Proposed: `AUTOMATED_UPSTREAM_EXTRACTION` — with mandatory qualification gate

**Bounded conditions** (`build_authoritative_input.py`):
- Unknown domain references are already recorded as `BOUNDED_CONDITION` — they don't abort, but they're tracked
- Same principle applies: unmappable Jira entities = BOUNDED_CONDITION, not silent drop

### Proposed: Jira Evidence Qualification Gate (L0)

A new L0 validation step that sits between the Jira API connector and `source_intake.py`. It assesses whether the extracted Jira data is structurally sufficient to produce meaningful intelligence.

```
Jira API
  → Jira Evidence Qualification Gate (NEW — L0)
      ├── Structural Completeness Assessment
      │     - Entity count sufficient? (minimum threshold)
      │     - Hierarchy present? (Epics → Stories, not flat)
      │     - Components/Labels used? (domain mapping possible)
      │
      ├── Relationship Density Assessment
      │     - Issue links present? (dependency/propagation edges)
      │     - Cross-project references? (cross-domain visibility)
      │     - Parent-child ratios? (structural depth)
      │
      ├── Temporal Coherence Assessment

      │     - Sprint completion patterns? (velocity consistency)
      │     - Creation-to-resolution distributions? (outlier detection)
      │
      ├── Mapping Viability Assessment
      │     - Can Jira entities map to semantic domains? (how many unmapped?)

      │

This turns the blame from "LENS is wrong" into "Jira data needs improvement."

## 15. Commercial Onboarding Motion

### 15.1 Engagement model — advisory-led structural intelligence

This is NOT a SaaS product with self-service onboarding. Not yet. And that's a feature, not a limitation.

**Why advisory-first is correct for Tier 1:**
1. **Evidence quality gate** — the advisory team curates evidence before the system processes it. This is the human quality gate that prevents "garbage in" (§8). Removing it prematurely destroys credibility.
2. **Ontology bridging** — mapping the buyer's planning ontology to structural domains requires human judgment. The Structural Domain Mapping Reference (§8) is an advisory artifact.
3. **Price justification** — advisory engagement justifies $15K-$40K per assessment. SaaS pricing at this stage would anchor too low.
4. **Relationship depth** — the buyer's first experience must be high-touch. The structural gap revelation is sensitive — it tells people their governance is partially blind. This needs human facilitation, not a product tour.
5. **Learning loop** — every advisory engagement produces intelligence about ontology patterns, evidence availability, and qualification distribution. This feeds connector design and automation. Advisory-first is also R&D.

**The advisory team is NOT a limitation to scale past.** It's a governed human-in-the-loop that ensures structural intelligence quality. Even at Tier 2 (connector-automated intake), the advisory relationship persists for interpretation guidance, governance consultation, and qualification depth.

### 15.3 The first conversation — discovery call structure

**Duration:** 30 minutes. NOT a demo. A diagnostic.

**Goal:** Determine if the buyer's program exhibits Execution Blindness symptoms and whether structural evidence is available for assessment.

**Structure:**

**Minutes 0-5 — Problem validation:**
- "How do you currently assess your program's structural health?"
- "Can you see cross-domain propagation risk in your current tools?"
- "If a foundational initiative slips, do you know which downstream programs are affected?"
- "How confident are you that your planning ontology represents your actual execution structure?"

**If the buyer answers these with some version of "we can't" or "we don't know" — they have Execution Blindness. Proceed.**

**Minutes 5-15 — Evidence availability assessment:**
- "What planning tools do you use?" (Jira, ADO, ServiceNow, spreadsheets)
- "How many programs/initiatives are in scope?"
- "Do you use Components in Jira? Issue links? Custom fields for domain or capability?"
- "Where does your architectural documentation live?"
- "Do you have access to dependency configurations, component structures, or evidence archives?"

**This determines feasibility.** If they have Jira with Components and Issue Links — high feasibility. If they have flat Jira with no hierarchy — lower feasibility, Q-03 likely, needs honest disclosure. If they have no structured evidence at all — advisory team assesses whether manual evidence curation is viable.

**Minutes 15-25 — Scope and expectations:**
- "We'd propose a structural assessment of [specific program/initiative]. First Structural Execution Visibility Report in 2-3 weeks."
- "We need access to [specific evidence types]. Our advisory team handles curation — no engineering effort required from your side."
- "The report will show you: structural topology, propagation risk, qualification posture, coverage gaps, and confidence boundaries."
- "It will also show you what it CAN'T assert — we disclose our limits, not just our findings."

**Minutes 25-30 — Next step:**
- Schedule the demo (if they haven't seen one)
- OR proceed directly to pilot assessment (if they've seen the demo / were referred)
- Introduce the advisory engagement scope and timeline
