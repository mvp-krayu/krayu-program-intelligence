## 1. Category and Brand

**Program Intelligence** — The discipline of applying governed structural intelligence to program execution assessment.

Program Intelligence is NOT:
- Project management ("we don't manage projects")
- Portfolio dashboarding ("we don't display status")
- AI code analysis ("we don't review code")
- DevOps tooling ("we don't deploy anything")
- Consulting methodology ("we don't advise — the structure speaks")

Program Intelligence IS:
- A new category of enterprise software
- Governed structural truth about execution reality
- Deterministic intelligence derived from evidence, not generated by AI
- The structural mirror that shows organizations what their tools cannot

## 2. What The Product Does

### 2.1 One-Sentence Definition

Signäl is a governed structural intelligence system that deterministically derives operational cognition from program evidence — showing organizations structural truth about execution that no existing tool, dashboard, or consultant can reveal.

### 2.2 The Core Capability (What Makes This Possible)

The product ingests evidence artifacts from program execution environments (repositories, planning tools, delivery telemetry), reconstructs the structural execution topology, and derives governed operational intelligence through a deterministic cognition pipeline.

**The moat:** AI participates inside bounded enterprise authority workflows without self-authorizing. The system THINKS deterministically (~80% of the pipeline) and COMMUNICATES with bounded AI assistance (~14%, under 75.x authority, disclosure-wrapped). The thinking is reproducible. The communication is governed. No other product architecture separates cognition from projection this way.

Five cognitive projection modes — not display variants, cognitive altitudes. Same structural truth, different depths.

## 5. Software Intelligence — The Commercial Module Model

SW-INTEL is the first Domain Module in the marketplace architecture. It proves the commercial module pattern.

### 5.1 What It Is

A domain cognition module that produces behavioral consequence intelligence from code structure analysis. It answers: "What does the structural topology tell us about execution risk?"

### 5.2 The 9-Slice Behavioral Cognition Ontology

| # | Slice | Class | What It Detects |
|---|-------|-------|----------------|
| 1 | Dependency Choke Point | Corridor | Single-path dependency bottlenecks |
| 2 | Propagation Asymmetry | Corridor | Unbalanced dependency flow creating cascade risk |
| 3 | Pressure Zone | Field | Areas of concentrated structural stress |
| 4 | Structural Mass Concentration | Field | Oversized components creating gravitational risk |
| 5 | Execution Fragility | Fragility | Components vulnerable to structural change |
| 6 | Execution Constriction | Flow | Bottlenecks constraining execution throughput |
| 7 | Structural Boundary Divergence | Drift | Boundaries drifting from intended architecture |
| 8 | Coupling Inertia | Coupling | Coupling patterns resisting architectural evolution |
| 9 | Compound Convergence | Composite | Multiple conditions converging on same structural area |

### 5.3 Commercial Model

- **Module OFF (default):** Teaser — consequence count, top consequence class, top severity, `requires_module: 'SOFTWARE_INTELLIGENCE'`. The user sees that intelligence EXISTS but must activate the module to access it.
- **Module ON (commercial activation):** Full 6-surface cognition view — severity, operational summary, consequence chain, expandable detail per slice. Full topology overlay with evidence-bound classification.
- **Module ABSENT:** No signal interpretations, no semantic domain registry entries. Module toggle not shown.

**Marketplace proof:** If a second domain module (M&A Intelligence, Portfolio Intelligence) can be added by:
1. Writing materializers for its cognition objects
2. Registering them with PICR
3. Adding a ProjectionConfig per consumer
4. Gating activation commercially

...without changing LENS core, PRE core, or PICP schema — then the marketplace architecture is proven.
