## 1. Executive Governance Summary

Software Intelligence is a governed software-operational consequence cognition runtime composed of evidence-bound behavioral consequence slices.

This document formalizes the slice doctrine — taxonomy, governance model, runtime placement, evidence contract, replay contract, promotion lifecycle, and persona projection rules — before any expansion of the slice ecosystem.

### What Software Intelligence IS

- Behavioral consequence cognition derived from structural topology
- Operational mechanics exposure through deterministic derivation
- Topology-to-behavior interpretation with evidence lineage
- Replayable consequence composition from governed evidence
- A domain cognition module within Program Intelligence that permeates all strata

### What Software Intelligence is NOT

- UI enhancement or narrative polish
- Dashboard feature expansion or generic software analytics
- Freeform AI interpretation or prompt-generated commentary
- A separate product or an alternate cognition runtime
- Better prose

## 2. Architectural Baseline

### Locked Three-Layer Architecture

The following architecture is CANONICAL (TERMINOLOGY_LOCK, 2026-05-27):

| Layer | Name | Scope |
|---|---|---|
| 1 | **PI Core** (~90%+) | Domain-agnostic cognition primitives — topology, centrality, signals, pressure, SQO, governance, replay, enrichment, personas. Truth layer. |
| 2 | **Orchestration-Agentic Runtime** (~9%) | Guided actions, SQO workflow orchestration, event-log-derived learning, action routing. |
| 3 | **Domain Cognition** (~1%) | Domain-specific operational interpretation — SignalSynthesisEngine, topology cognition projection, consequence compilation, module-specific rendering. |

This architecture is not modified by this document.

### Derivation Flow Model (L1–L4)

Within the locked three-layer architecture, Software Intelligence derivation follows a control-flow sequence:

| Stage | Name | Layer Ownership | Description |
|---|---|---|---|
| L1 | Structural Derivation | PI Core (Layer 1) | Signal families (PSIG/DPSIG/ISIG), topology metrics, evidence blocks, propagation analysis |
| L2 | Consequence Cognition | Domain Cognition (Layer 3) | SignalSynthesisEngine conditions, ConsequenceCompiler consequence types, topology cognition slices |
| L3 | Governed Projection Composition | Domain Cognition (Layer 3) | Persona-specific projections (forBoardroom, forBalanced), BALANCED composition |
| L4 | Persona Rendering | LENS (cross-layer) | RepresentationField, ExecutiveInterpretation, SupportRail — rendering only |

**Relationship:** L1–L4 describes the derivation flow of a single datum from evidence to screen. The Three-Layer Architecture describes ownership. L1 maps to Layer 1. L2-L3 map to Layer 3. L4 maps to the LENS projection surface. These models are complementary — one traces data flow, the other defines authority.

**Binding constraint:** Truth derivation occurs at L1-L2 only. L3 composes, sequences, frames, and paces existing truth. L4 renders. No truth generation at L3 or L4.

### Where Slices Belong

SW-INTEL consequence slices are **L2 objects** — they belong to the Consequence Cognition stage, owned by Domain Cognition (Layer 3).

Slices are derived from L1 structural evidence. They are consumed by L3 projection composition. They are rendered at L4. They are never generated at L3 or L4.

## 3. Existing Vocabulary Hierarchy

The SW-INTEL cognition pipeline already contains four distinct concept layers. This hierarchy is operational and must be preserved:

| Layer | Concept | Location | Count | Description |
|---|---|---|---|---|
| Evidence | **Signal families** | PI Core (PSIG/DPSIG/ISIG) | 8 signals | Raw structural measurements from pipeline artifacts |
| Condition synthesis | **Conditions** | SignalSynthesisEngine | 6 primitive + 1 composite | Deterministic compilation of signals into operational condition types |
| Cognition derivation | **Surfaces** | SoftwareIntelligenceProjectionAdapter | 6 surfaces | Multi-source cognition synthesis (DELIVERY_FRAGILITY, COORDINATION_SATURATION, etc.) |
| Consequence compilation | **Consequence types** | ConsequenceCompiler | 8 primitive + 3 combination | Operational implication objects derived from conditions |
| Topology projection | **Topology cognition slices** | Topology Cognition Language | 4 MVP slices | Visual/semantic overlays projecting conditions onto structural topology |

**Key distinction:** Each layer transforms the output of the layer above. Signals become conditions. Conditions feed surfaces and consequence types. Consequence types are projected through personas. Topology cognition slices project conditions visually.

SW-INTEL consequence slices (the subject of this document) do NOT collapse these layers. They are a **cross-cutting classification** that identifies governed behavioral patterns across the existing hierarchy.

## 4. Terminology Evolution

### Controlled G1 Terminology Change

The term "slice" currently has a locked narrow definition in TERMINOLOGY_LOCK:

> **Topology Cognition Language:** "The 4-slice MVP visual and semantic language through which LENS projects operational conditions onto the structural topology."

This document introduces a broader concept while preserving the existing definition:

| Term | Scope | Definition |
|---|---|---|
| **Topology cognition slice** (existing, locked) | Topology projection | A visual/semantic overlay projecting an operational condition onto the structural topology graph. 4 MVP slices: Dependency Choke Point, Propagation Asymmetry, Pressure Zone, Structural Mass. Categorized as corridor cognition (Slices 1-2) or field/state cognition (Slices 3-4). |
| **SW-INTEL consequence slice** (new, introduced by this document) | Consequence cognition | A canonical, evidence-bound, replayable representation of a software-operational behavioral pattern derived from governed structural and/or consequence evidence. The broader class that encompasses topology cognition slices as its projection specimens. |

**Relationship:** Topology cognition slices are the **projection instances** of SW-INTEL consequence slices. Every topology cognition slice corresponds to an underlying consequence slice that defines the behavioral pattern, evidence contract, and replay guarantees. The topology cognition slice is how it appears on the topology graph. The consequence slice is the governed cognition object.

**The existing 4 topology cognition slices are the first 4 SW-INTEL consequence slices projected as topology overlays.** This document does not rename them — it identifies the broader class they belong to.

## 5. Slice Definition

An SW-INTEL consequence slice is a canonical, evidence-bound, replayable representation of a software-operational behavioral pattern derived from governed structural and/or consequence evidence.

### Qualification Criteria

Every valid consequence slice must answer all of the following:

| # | Question | Required |
|---|---|---|
| 1 | What behavioral pattern does it represent? | Named pattern with deterministic definition |
| 2 | Which evidence objects activate it? | Explicit evidence artifact references |
| 3 | Which structural/topological mechanics does it expose? | Topology behavior description |
| 4 | Which operational consequence does it qualify? | Mapped to consequence type or candidate |
| 5 | Which runtime layer owns derivation? | Must be L2 (Consequence Cognition) |
| 6 | Which consequence object may consume it? | ConsequenceCompiler mapping or NONE |
| 7 | Which personas may project it? | Per persona projection matrix (§17) |
| 8 | Which evidence lineage proves it? | Traceable derivation chain |
| 9 | Which replay guarantees exist? | Same evidence → same slice output |
| 10 | Which governance constraints apply? | Evidence classification, confidence basis |

A candidate that cannot answer all 10 is not yet a slice. It may be a CANDIDATE (§13).

## 6. Non-Slice Definition

The following are explicitly NOT SW-INTEL consequence slices:

- **Prose paragraph** — narrative text, however operational, is projection not cognition
- **Visual decoration** — topology styling without evidence binding
- **UI card or component** — rendering artifact, not cognition object
- **Persona-specific insight** — personas consume slices, they do not create them
- **Raw metric** — metrics are L1 evidence, not L2 behavioral patterns
- **Ungoverned heuristic** — patterns without deterministic derivation rules
- **Prompt-generated interpretation** — AI inference without evidence binding
- **Narrative label** — display text is L4 rendering, not L2 cognition
- **Duplicate consequence compiler** — parallel derivation paths are prohibited
- **Alternate semantic derivation** — only one canonical truth path exists

## 8. Maturity Classification

### FOUNDATIONAL Slices

Characteristics:
- Deterministic — same evidence produces same output
- Directly evidence-bound — single evidence source or small evidence set
- Topology/metric-derived — structural facts, not inferred patterns
- Low interpretive risk — minimal judgment in derivation
- Suitable for runtime adoption now

Current foundational slices (5):
1. Propagation Asymmetry (Topology Slice 2)
2. Dependency Choke Point (Topology Slice 1)
3. Structural Mass Concentration (Topology Slice 4)
4. Pressure Zone Convergence (Topology Slice 3)
5. Import Pressure Concentration (no topology projection yet)

### ADVANCED Slices

Characteristics:
- Composite — require multiple evidence sources
- May require temporal comparison (multi-run)
- May require interaction between slice outputs
- Higher governance burden — stronger replay/provenance obligations
- Must not be implemented before foundational governance is stable

All CANDIDATE slices in §7 are classified ADVANCED until evidence proves otherwise.

## 9. Slice Evidence Contract

Every SW-INTEL consequence slice must satisfy the following evidence requirements:

| Requirement | Description |
|---|---|
| Source artifacts | Named pipeline artifacts that provide raw evidence |
| Structural inputs | Specific structural measurements consumed |
| Signal inputs | PSIG/DPSIG/ISIG signal IDs that contribute |
| Consequence inputs | Consequence type IDs consumed (if any) |
| Temporal inputs | Multi-run comparison data (if required — ADVANCED only) |
| Activation rule | Deterministic condition that activates the slice |
| Confidence basis | Evidence classification of the derivation |
| Evidence identifiers | Specific artifact IDs, signal IDs, domain IDs |
| Replay path | Exact sequence to reproduce the slice from stored evidence |

**Governing rule:** No evidence → no slice. A slice with insufficient evidence is suppressed, never fabricated.

## 10. Slice Replay / Provenance Contract

Every slice output must be reproducible from stored governed evidence.

### Conceptual Slice Output Schema

This is a governance contract, not an implementation schema. Implementation requires a separate stream.

```
slice_id               — unique identifier
slice_name             — human-readable name
slice_type             — ontology class (A-E)
slice_class            — FOUNDATIONAL | ADVANCED
slice_maturity         — promotion lifecycle state (§13)
activation_status      — ACTIVE | INACTIVE | SUPPRESSED
behavioral_pattern     — named pattern description
operational_mechanics  — topology behavior exposed
operational_consequence — consequence type mapping (if any)
evidence_refs          — list of evidence artifact references
source_artifact_refs   — pipeline artifact identifiers
derivation_rule_ref    — reference to deterministic derivation rule
runtime_layer          — L2 (Consequence Cognition)
consumable_by          — which L3 compositions may consume this
projectable_in         — persona projection permissions
projection_constraints — per-persona rendering constraints
confidence_basis       — evidence classification
replay_status          — REPRODUCIBLE | PARTIAL | UNAVAILABLE
provenance_status      — GOVERNED | UNGOVERNED
governance_status      — COMPLIANT | WARNING | VIOLATION
warnings               — governance warnings (if any)
```

**Governing rule:** No slice may depend on hidden prompt reasoning or unlogged AI inference. Every field must be deterministically derivable from governed evidence.

## 12. Relationship to Consequence Compiler

### What slices may do

- Provide behavioral consequence objects to the consequence pipeline
- Enrich consequence cognition with additional behavioral patterns
- Be consumed by ConsequenceCompiler persona projections (forBoardroom, forBalanced)
- Support operational interpretation sequencing in BALANCED

### What slices may NOT do

- Replace ConsequenceCompiler
- Fork consequence truth
- Generate competing consequence objects
- Derive persona-specific truth
- Bypass canonical consequence cognition

### Current implementation reality

The ConsequenceCompiler already consumes the 5 foundational slices indirectly — through SignalSynthesisEngine conditions. The derivation chain is:

```
Signals (L1) → Conditions (L2) → Consequence Types (L2) → Persona Projections (L3) → Rendering (L4)
```

Topology cognition slices project conditions visually at L4. Consequence slices (this document's subject) classify the behavioral patterns at L2. Both consume the same L1 evidence through different derivation paths.

## 13. Slice Promotion Lifecycle

| State | Authority | Description |
|---|---|---|
| **CANDIDATE** | None — concept only | Behavioral pattern identified. No evidence binding. No derivation rule. Documentation only. |
| **SPECIMEN** | Evidence observed | Deterministic evidence pattern exists. Testable against known artifact set. Not yet composable. No runtime authority. |
| **FOUNDATIONAL** | Evidence-bound | Replayable. Deterministic. Lineage-traceable. Allowed for runtime composition. |
| **COMPOSABLE** | Consequence-ready | Can be safely consumed by ConsequenceCompiler. Can participate in BALANCED composition. Has persona projection rules. |
| **CERTIFIED** | Production-grade | Tested across multiple clients/specimens. Replay-proven. Governance-approved. Stable for product surface. |
| **DEPRECATED** | Preserved-only | Superseded or invalidated. Preserved for replay/audit but no longer active in runtime. |

**Promotion rules:**
- Each promotion requires explicit evidence
- No skip-level promotion (CANDIDATE cannot become COMPOSABLE directly)
- Demotion is permitted with governance justification
- DEPRECATED slices remain in the taxonomy for lineage

- Deriving SW-INTEL semantics inside React components
- Deriving SW-INTEL semantics inside NarrativePrimitives or ZoneComposer
- Deriving SW-INTEL semantics from fullReport directly in projection layers (L3/L4)

- Creating a new SW-INTEL report generator that bypasses consequence cognition
- Creating prompt-based slice interpretation

- One canonical cognition substrate
- One consequence truth path
- Projection-only behavior at L3 and L4
- Explicit feature gating (SW-INTEL ON/OFF)

## 16. Composition Model

### Rules

- Slices may compose into higher-order consequence cognition
- Composition must be governed (deterministic rules, not inference)
- Composition must preserve evidence refs from all contributing slices
- Composite slices require stronger replay/provenance obligations than primitives
- Conflicting slices must be surfaced as governed tension (§16.1), not silently merged
- Composition occurs at L2, never at L3 or L4

### Existing composition specimen

ConsequenceCompiler combination patterns are the current composition model:
- AMPLIFIED_DEPENDENCY_FRAGILITY (pressure + choke point co-location)
- STRUCTURAL_GRAVITY_WELL (mass + pressure co-location)
- SYSTEMIC_OPERATIONAL_FRAGILITY (3+ independent conditions converging)

These demonstrate governed composition: deterministic rules, evidence from contributing primitives preserved, new consequence objects produced only when convergence conditions are met.

### 16.1 Conflict / Tension Model

When slices produce contradictory, weak, or overlapping signals:

| Condition | Handling |
|---|---|
| Contradictory slice signals | Surface as governed tension with evidence refs from both sides |
| Weak evidence | Downgrade confidence, never suppress silently |
| Partial activation | Report partial state, do not force full activation |
| Competing operational implications | Present both with evidence weight, do not resolve narratively |
| Overlapping topology areas | Classify overlap type (reinforcing vs competing) |
| Stale evidence | Flag staleness, do not use stale evidence as current authority |
| Unavailable replay path | Mark replay_status: UNAVAILABLE, suppress from CERTIFIED projection |

**Governing rule:** Do not resolve conflicts narratively. Represent them as governed tension with evidence references.

## 17. Commercial Boundary

### PI-Core

Governed structural operational intelligence. Domain-agnostic cognition primitives. The truth layer.

### SW-INTEL

Software-operational consequence intelligence. A domain cognition module that exposes behavioral mechanics PI-Core does not specialize in.

**SW-INTEL ON:** PI-Core enhanced with consequence cognition overlays.
**SW-INTEL OFF:** Pure PI-Core structural intelligence. No SW-INTEL code executes.

**Correct distinction:** SW-INTEL exposes behavioral software-operational mechanics that PI-Core does not specialize in. It does not replace, weaken, or compete with PI-Core. PI-Core is the foundation. SW-INTEL is the domain-specific interpretation layer.
