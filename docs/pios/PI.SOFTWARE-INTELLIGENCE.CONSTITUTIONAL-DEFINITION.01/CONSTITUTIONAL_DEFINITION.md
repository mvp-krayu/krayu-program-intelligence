## 1. What Software Intelligence IS

Software Intelligence is a **domain cognition module** attached to the PI Spine. It provides operational software cognition: the interpretation of PI cognition primitives through software-system semantics.

Software Intelligence transforms PI Core outputs from structurally correct but operationally abstract intelligence into domain-specific operational meaning that executives, engineers, and analysts can act on.

Without Software Intelligence:
- Pressure is "stress somewhere"
- Propagation is "chain of connected nodes"
- Centrality is "a file has many edges"
- Topology is "clusters and connections"

With Software Intelligence:
- Pressure becomes "deployment coordination instability at the runtime orchestration layer"
- Propagation becomes "change in Platform Infrastructure propagates to 5 downstream service domains"
- Centrality becomes "orchestration bottleneck — globals.py coordinates 12 downstream modules"
- Topology becomes "backend/frontend architectural separation with API boundary coupling"

Software Intelligence is the first instance of a general-purpose architectural pattern: **domain cognition modules**. This pattern — a replaceable interpretation layer that gives PI Core domain-specific operational meaning — is the foundation of Marketplace.

## 2. What Software Intelligence is NOT

- **NOT code metrics.** Lines of code, cyclomatic complexity, code coverage percentages are tool outputs. Software Intelligence is operational cognition derived from structural evidence.
- **NOT GitHub analytics.** Commit frequency, PR velocity, contributor graphs are activity metrics. Software Intelligence operates on structural topology, not activity.
- **NOT repository visualization.** Folder trees, dependency graphs, file browsers are display tools. Software Intelligence interprets structural relationships through operational semantics.
- **NOT taxonomy mapping.** Renaming folders to prettier labels ("src/backend" → "Backend Services") is cosmetic. Software Intelligence derives operational meaning from structural evidence — role abstractions trace to structural entities with bidirectional mappings.
- **NOT AI commentary.** Generative interpretation, freeform analysis, or speculative assessment. All Software Intelligence outputs trace to structural evidence through deterministic derivation.
- **NOT a separate product.** Software Intelligence is a module WITHIN Program Intelligence. It permeates all strata — it is not an attachment point, a plugin, or an optional add-on.

## 4. The 10 Software Intelligence Cognition Functions

Each cognition function transforms a PI Core primitive into software-domain operational intelligence. The function receives PI Core output, applies software-operational interpretation, and produces domain-specific operational output.

### CF-01: Pressure Interpretation

**PI Core primitive:** Pressure concentration detection — zones, conditions, compound activation
**SW-Intel interpretation:** Pressure zones become typed operational risks:
- Deployment fragility — pressure in release/deployment corridors
- Orchestration overload — pressure concentrating at coordination spines
- Integration saturation — pressure at API boundary or cross-domain coupling points
- Test authority gap — pressure in validation coverage asymmetry zones

**Operational output:** Typed pressure zone with operational risk classification, domain-specific severity language, and consequence framing ("deployment coordination instability" vs "pressure in Zone A")

### CF-02: Execution Corridor Detection

**PI Core primitive:** Propagation chain detection — directional flow through topology
**SW-Intel interpretation:** Propagation chains become delivery-critical paths:
- Delivery corridors — chains through which changes must propagate to reach production
- Orchestration corridors — chains through which runtime coordination flows
- Runtime dependency spines — chains of module-level structural dependency

**Operational output:** Named execution corridors with domain-specific role assignments (ORIGIN becomes "change source domain," PASS_THROUGH becomes "coordination relay," RECEIVER becomes "deployment target")

### CF-03: Coordination Spine Detection

**PI Core primitive:** Structural centrality — 40.3c normalized metrics, structural role classification
**SW-Intel interpretation:** Centrality patterns become operational coordination roles:
- Orchestration hubs — RUNTIME_SPINE files that coordinate cross-module execution
- Governance bottlenecks — files with disproportionate structural authority
- Configuration centers — UTILITY_HUB files providing shared operational state
- Coupling concentrators — files where cross-domain dependencies converge

**Operational output:** Named coordination spines with operational consequence language ("globals.py is an orchestration hub — 12 downstream modules coordinate through this single point, creating deployment coupling risk")

### CF-04: Validation Intelligence

**PI Core primitive:** Coverage asymmetry, grounding ratio, qualification blockers
**SW-Intel interpretation:** Coverage gaps become operational risk areas:
- Test authority gaps — domains with structural presence but no testing coverage
- Replay instability — areas where deterministic revalidation cannot confirm behavior
- Qualification fragility — domains where evidence is thin relative to structural weight

**Operational output:** Per-domain validation posture with operational risk language ("Platform Infrastructure carries 35% of structural mass but has WEAK test authority — qualification fragility risk")

### CF-05: Release/Deployment Cognition

**PI Core primitive:** Propagation patterns, structural coupling, blast radius
**SW-Intel interpretation:** Propagation measurements become deployment risk intelligence:
- Deployment risk corridors — paths through which a single change propagates to multiple production surfaces
- Change amplification chains — structural paths where small changes produce disproportionate downstream impact
- Release isolation assessment — whether domains can be deployed independently

**Operational output:** Deployment risk assessment per domain with blast-radius language ("Change in Platform Infrastructure propagates through 5 downstream domains — deployment coordination required across 3 service boundaries")

### CF-06: Execution Governance

**PI Core primitive:** Governance lifecycle, SQO state transitions, authority events
**SW-Intel interpretation:** Governance measurements become execution maturity intelligence:
- Governance enforcement density — how thoroughly governance is exercised across the software lifecycle
- Execution drift detection — divergence between governance claims and structural evidence
- Policy concentration — where governance decisions cluster vs where they are absent

**Operational output:** Execution maturity assessment with software-specific governance language ("Governance friction surfaced in 3 domains — contested propositions indicate structural disagreement between automated derivation and operator assessment")

### CF-07: Topology Role Abstraction

**PI Core primitive:** Structural topology — 40.2/40.3/40.4 evidence chain, structural roles from 40.3c
**SW-Intel interpretation:** Raw structural paths become operational layers:
- `src/backend/core/` → "Runtime Orchestration Layer"
- `src/api/` → "API Boundary Layer"
- `src/models/` → "Data Authority Layer"
- Structural roles (RUNTIME_SPINE, UTILITY_HUB, etc.) → operational role names

**Operational output:** `role_abstractions` — a bidirectional mapping from structural entity IDs to operational domain names. Every abstraction traces to a structural source. No orphaned abstractions.

### CF-08: Operational Attention Routing

**PI Core primitive:** Multi-signal activation — PSIG, DPSIG, ISIG compound patterns
**SW-Intel interpretation:** Signal compound patterns become operational attention priorities:
- When 2+ signal families activate in the same zone → "pressure corridor" (executive attention required)
- When ISIG activates without PSIG → "file-level structural risk not visible at architecture level" (engineering attention)
- When DPSIG shows asymmetry → "architectural imbalance requiring structural review" (CTO attention)

**Operational output:** Attention signals with audience-specific routing ("Engineering leadership: Import Hub Pressure 35.3 HIGH in Platform Infrastructure — deployment coordination cost concentrates at file level, invisible at architectural overview")

### CF-09: Pressure-Aware Topology Projection

**PI Core primitive:** Static topology + pressure zone state
**SW-Intel interpretation:** Topology becomes operationally reactive:
- Pressure zones change domain visual prominence (pressured domains are structurally foregrounded)
- Execution corridors illuminate under pressure (the path pressure takes through the system)
- Coupling edges gain operational type labels (structural dependency vs runtime coordination vs test authority)

**Operational output:** Pressure-aware software topology projection — the SVG/topology becomes "operational software cognition topology" not "semantic cluster visualization." Zone boundaries, member visibility, condition decomposition rendered through software-operational semantics.

### CF-10: Engineering Qualification Cognition

**PI Core primitive:** SQO state machine + governance lifecycle + authority corridor model
**SW-Intel interpretation:** Qualification measurements become deployment readiness intelligence:
- S-state → deployment readiness tier ("S2 GOVERNED = operationally qualified with known debt")
- Qualification blockers → deployment constraints ("NO_SEMANTIC_DERIVATION = domain structure not yet validated for production governance")
- Governance maturity → operational confidence level ("Governed review exercised across 85 propositions — governance friction indicates structural scrutiny, not dysfunction")

**Operational output:** Engineering qualification posture with deployment-specific language, advancement path, and operational consequence framing

## 8. What Software Intelligence is NOT (Expanded Prohibitions)

### Prohibition 1: No Taxonomy Mapping

Taxonomy mapping (folder → prettier label) is cosmetic renaming. Software Intelligence derives operational meaning from structural evidence. A taxonomy map says "`src/backend/` = Backend." Software Intelligence says "`src/backend/core/globals.py` is an orchestration hub coordinating 12 downstream modules — this file's structural centrality creates deployment coupling risk."

The difference is operational cognition vs label assignment. Labels inform nothing. Operational cognition informs decisions.

### Prohibition 2: No AI Commentary

Software Intelligence is not generative interpretation. Every SW-Intel output traces to structural evidence through deterministic derivation. There is no "AI thinks" or "AI suggests" in Software Intelligence. There is "structural evidence shows" and "operational interpretation is."

### Prohibition 3: No Code Metrics Aggregation

Lines of code, cyclomatic complexity, code coverage, code churn — these are tool metrics. Software Intelligence operates at a different abstraction level: structural topology cognition. Code metrics measure individual file characteristics. Software Intelligence measures structural relationships, coordination patterns, and operational topology.

### Prohibition 4: No Repository Visualization Tooling

File browsers, dependency graphs, folder trees — these are visualization tools. Software Intelligence is not a visualization layer. It is a cognition layer that produces operational meaning. The visualization is LENS. The cognition is Software Intelligence. The evidence is PI Core.

### Prohibition 5: No Separate Product

Software Intelligence is not a standalone product. It is a module WITHIN Program Intelligence. It does not have its own UI, its own API, its own deployment. It permeates the existing PI stack, enriching every stratum with domain-specific operational meaning.
