## 3. Impact Chain for Client C

If a hypothetical Client C has PATH A completed (structural scan, CEU grounding, 13+ DOMs) but NO semantic ontology:

```
semantic_topology_model.json ABSENT
  → SemanticActorHydrator has no DOMAINs to hydrate
    → semanticCrosswalk is EMPTY (no DOMAINs to bridge)
      → ReconciliationCorrespondenceCompiler produces NULL output
        → grounding_ratio = undefined
          → Q-class = undefined (Q-00 or unqualified)
            → LENS semantic zones render EMPTY
              → SQO qualification stays at S1 (structural-only)
```

**What DOES work without PATH B:**
- PATH A structural topology renders (all 13+ DOMs visible)
- Structural signals compute (from DOM-level analysis)
- GAUGE score computes (from structural coverage)
- Structural zones render (IntelligenceField, StructuralTopologyZone)

- Signal assessment works

**What DOES NOT work without PATH B:**
- Semantic domain labels (DOMAIN-01..DOMAIN-N)

- Reconciliation correspondence
- Grounding ratio computation
- Q-class determination
- Semantic zones (ReconciliationAwarenessZone, SemanticTrustPostureZone)
- Domain enrichment rationale
- Full guided interrogation (some queries depend on semantic data)

## 4. The Irreducible Boundary

**Where automation ends and human judgment begins:**

The irreducible boundary is **semantic ontology authoring** — the identification and classification of business DOMAINs, capabilities, and components from evidence analysis.

PATH A is fully automated: source archive → structural scan → CEU grounding → DOM compression. Given evidence, the engine produces DOMs deterministically.

PATH B requires human judgment: an analyst must examine evidence, identify semantic business domains (e.g., "Edge Data Acquisition," "Fleet Core Operations"), classify capabilities within domains, and map components to capabilities. This classification cannot be automated from source code structure alone because:

1. **Business domain boundaries are semantic, not structural.** A module named `VehiclesModule` could belong to "Fleet Management," "Asset Tracking," or "Logistics" depending on the client's business context. Code structure doesn't encode business intent.

2. **Capability classification requires domain expertise.** Whether a module represents a "CORE" capability vs a "SUPPORTING" one depends on business strategy, not code architecture.

3. **Component naming is arbitrary.** The same structural pattern (a NestJS module) carries different semantic weight in different business contexts. Naming conventions don't generalize.

4. **Cross-domain relationships are judgment calls.** Whether COMP-25 (OtaModule) cross-cuts into DOM-01 is a semantic decision about business function, not a structural fact derivable from code.

The CSR (Client Semantic Registry) specification addresses this by defining the governed process through which this human judgment is captured, structured, and version-controlled.

## 5. BlueEdge-Specific Assumptions That MUST NOT Leak Into Generic CSR

The following are BlueEdge runtime facts. They MUST NOT become implicit defaults, schema constraints, or architectural assumptions in the generic Client Semantic Registry or onboarding substrate:

| Assumption | BlueEdge Value | Why It Must Not Leak |
|---|---|---|
| DOMAIN count | 17 | A new client may have 5, 30, or 100 DOMAINs. The CSR schema must be count-agnostic. |
| Capability count | 42 | Varies by client complexity. No fixed cardinality. |
| Component count | 89 | Varies by client evidence size. No fixed cardinality. |
| DOM-09 irresolvability | 1 DOM → 6+ DOMAINs | DOM-09 is a structural reality of BlueEdge's backend_modules directory. Other clients may have zero irresolvable DOMs, or many. The irresolvability detection algorithm must be general, not calibrated to DOM-09. |
| Grounding ratio | 4/17 DOMs→DOMAINs (23.5%) | This ratio depends on crosswalk quality and structural alignment. A new client's ratio could be 0% (no crosswalk) or 100% (perfect alignment). The ratio is an OUTPUT of the process, not a TARGET. |
| Q-class | Q-02 (Partial Grounding) | Q-02 is BlueEdge's current runtime state. A new client starts at Q-00 (no qualification). Q-class is computed from data, not prescribed. |
| DOMAIN names | "Edge Data Acquisition," "Fleet Core Operations," etc. | These are BlueEdge's business domains. A fintech client would have "Payment Processing," "Risk Engine," etc. No domain names should appear as defaults. |
| DOMAIN types | FUNCTIONAL, INFRASTRUCTURE, CROSS-CUTTING, OPERATIONAL, INTEGRATION | The type taxonomy MAY generalize, but MUST be validated against multiple clients before being treated as universal. |
| Reconciliation confidence thresholds | 0.90 / 0.65 / 0.50 for L5/L4/L3 | Calibrated against BlueEdge crosswalk confidence distribution. May need adjustment for clients with different structural characteristics. |
