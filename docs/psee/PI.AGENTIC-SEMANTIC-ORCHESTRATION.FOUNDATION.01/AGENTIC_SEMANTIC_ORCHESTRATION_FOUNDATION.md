## 3. Correct Role of LLMs

### 3.1 Framing

LLMs are the most capable natural language systems available. They reason, summarize, explain, generate prose, answer questions, and synthesize patterns across large contexts. None of those capabilities require them to own structural truth.

The question is not "can LLMs do more?" They can. The question is "what should they own in a governed platform?" The answer is bounded and precise.

### 3.2 LLM Capability Matrix

| Capability | LLM Permitted | Governance Condition |
|---|---|---|
| Explain why a cluster has high CPI | YES | Must cite the specific CPI value and derivation evidence |
| Summarize escalation propagation across clusters | YES | Must cite specific signal activations |
| Narrate the executive risk picture for a client | YES | Requires executive_rendering_allowed = true |
| Generate comparison narrative (historical pressure change) | YES | Must cite both snapshot states; no invented baselines |
| Suggest probable remediation strategies | YES | Must be framed as suggestion, not diagnosis |
| Answer executive questions about topology | YES | Must ground answers in committed evidence |
| Generate governance summaries | YES | Must cite source documents |
| Classify a domain as HIGH-risk | NO | Classification is owned by _classify_dpsig_readiness_state |
| Modify a CPI threshold | NO | Thresholds are locked; SCRIPT_VERSION increment required |
| Override a DIAGNOSTIC_ONLY readiness state | NO | Gate is authoritative; no override path |
| Generate a new signal activation | NO | Signals are derived by deterministic scripts only |
| Infer cluster topology structure | NO | Topology is sovereign; no inference permitted |
| Decide which nodes are under pressure | NO | Pressure is computed by 75.x; RUN_RELATIVE_OUTLIER sovereign |
| Store reasoning as permanent platform truth | NO | LLM outputs are not governed artifacts unless explicitly captured |

### 3.3 Authority Boundary Definition

**LLMs operate in the interpretation layer.** They are downstream consumers of governed structural evidence. They receive:

- Signal values (CPI, CFA, PSIG-001..006) — read-only
- Readiness state (EXECUTIVE_READY / DIAGNOSTIC_ONLY / etc.) — read-only
- Domain grounding classification (EXACT / STRONG / PARTIAL / NONE) — read-only
- Aliased cluster/domain names (when readiness allows) — read-only
- Semantic topology model — read-only

They produce:

- Natural language explanations
- Executive narratives
- Comparative analyses
- Remediation suggestions
- Diagnostic prose

They never produce:

- Signal values
- Topology modifications
- Readiness classifications
- Threshold adjustments
- Activation decisions

### 3.4 Prohibited Authority Domains

LLMs are permanently prohibited from owning:

| Domain | Why Prohibited |
|---|---|
| Structural truth generation | Truth is derived by deterministic scripts from committed evidence |
| Signal activation | Activation is owned by 75.x (PSIG) and derive_relational_signals.py (DPSIG) |
| Topology mutation | Topology is sovereign and replay-safe; mutation breaks determinism |
| Threshold modification | Thresholds are locked; changes require SCRIPT_VERSION + governance stream |
| Readiness classification | Gate is the false-positive containment mechanism; LLM override = false-positive risk |
| Semantic authority | Semantic governance was formally closed; reopen conditions not met |
| Platform memory | LLM session state is not a governed artifact |
| Replay identity | LLM outputs must not be cited as evidence in replay validation |

### 3.5 Governance-Safe Participation Model

LLMs participate safely when:

1. All inputs are governed artifacts (committed evidence, not inferred state)
2. All outputs are explicitly labeled as interpretation, not classification
3. Outputs are either ephemeral (session-only) or captured as governed artifacts with full lineage
4. No output can mutate Layer 1 or Layer 2 platform state
5. Executive rendering is gated by executive_rendering_allowed — LLMs do not override this
6. Prompt inputs are traceable (for semantic replay governance — Section 9)

**Governing rule:** LLMs interpret deterministic evidence. LLMs do NOT generate structural truth.

## 4. Agentic AI Positioning

### 4.1 What Agents Are

Agents in this architecture are LLM-driven systems that combine natural language reasoning with tool use — the ability to query structured data, retrieve evidence, navigate topology, and chain operations across multiple steps.

They are not autonomous decision-makers for platform state. They are governed reasoning systems that operate on top of locked structural evidence.

| Threshold adjustment | GOVERNANCE_RESTRICTED | Permanently prohibited; requires governance stream |

### 4.3 Orchestration Boundaries

Agents may orchestrate across:
- Multiple evidence queries (reading different committed artifacts)
- Multi-step reasoning chains (interpreting cluster → domain → zone relationships)
- Multiple output formats (narrative + structured summary + executive brief)
- Historical comparison (if committed snapshots exist)

Agents may NOT orchestrate across:
- Signal derivation (orchestration cannot initiate or modify signal computation)
- Topology mutation (agents cannot request topology changes)
- Readiness gate state (agents cannot request gate re-evaluation outside the gate contract)
- Client-specific evidence fabrication (agents cannot invent evidence for missing clients)

### 4.4 Structural Authority Exclusions

The following are permanently excluded from agent authority:

- `vault/signal_registry.json` — write access forbidden
- `binding_envelope.json` — write access forbidden; PSIG sovereign
- `canonical_topology.json` — write access forbidden; structural truth
- `_classify_dpsig_readiness_state()` — agents cannot modify or bypass this function
- DPSIG thresholds (CPI_HIGH=5.0, CFA_DOMINANT=0.60) — agents cannot suggest threshold changes as implementation actions

### 4.5 Replay-Safe Orchestration Model

Orchestration is replay-safe when:

1. All agent inputs are committed, versioned artifacts
2. The agent's reasoning chain can be reconstructed from the same inputs
3. Agent outputs are either ephemeral (session-only) or captured with prompt + evidence lineage
4. No agent output is treated as platform truth without explicit governance promotion
5. Agent operations do not modify the evidence they reason over

Orchestration is NOT replay-safe when:
- Agents read from non-committed state (in-memory cache, session variables)
- Agent outputs overwrite committed artifacts without governance stream
- Agent reasoning cannot be reproduced from the committed evidence set

## 5. Semantic Interpretation Layer

### 5.1 Layer Responsibilities

The semantic interpretation layer bridges deterministic structural evidence and natural language intelligence. It exists to answer questions like:

- "Explain why CLU-04 (Platform Infrastructure) is showing elevated structural concentration."
- "Summarize how escalation propagates from DOMAIN-09 to DOMAIN-10."
- "Compare the pressure topology between the current run and the previous run."
- "Generate the executive risk narrative for the BlueEdge platform."
- "Identify organizational coupling themes across the five semantic clusters."
- "Suggest remediation strategies for the CPI=2.1176 finding in backend_modules."
- "Explain the probable structural contributors to CLUSTER_PRESSURE_ELEVATED."

Each answer requires:
1. Reading committed structural evidence (Layer 1–2)
2. Applying semantic grounding knowledge (Layer 3)
3. Generating natural language interpretation (Layer 4)
4. Rendering within cognitive projection constraints (Layer 5)

### 5.2 Semantic Confidence Governance

Semantic interpretation is confidence-gated. Interpretation may only claim as much authority as the underlying grounding warrants.

| Grounding Lineage | Interpretation Authority | LLM Language Permission |
|---|---|---|
| EXACT (confidence ≥ 0.90) | Full attribution | "The Edge Data Acquisition domain shows..." |
| STRONG (confidence 0.70–0.89) | Qualified attribution | "The Platform Infrastructure domain likely shows..." |
| PARTIAL (confidence 0.50–0.69) | Soft attribution with explicit caveat | "The Event-Driven Architecture domain (validate with engineering) appears to show..." |
| NONE (confidence < 0.50) | Structural label only | "Domain cluster DOMAIN-05 shows..." |
| inference_prohibition=True | Blocked entirely | Raw identifier only; no interpretation |

LLM outputs that claim attribution beyond their grounding lineage are a governance violation. The semantic interpretation layer must enforce confidence-bounded language.

### 5.3 Grounding Inheritance Rules

1. An LLM-generated narrative inherits the grounding state of its source evidence
2. A narrative citing EXACT-lineage domains may use full business language
3. A narrative citing NONE-lineage domains must use structural labels only
4. A narrative citing mixed-lineage domains must apply per-domain language gating
5. A narrative generated for a DIAGNOSTIC_ONLY client must use no business attribution regardless of domain lineage
6. Grounding inheritance is NOT transitive by proximity — adjacent domains do not inherit each other's confidence

### 5.4 Semantic Evidence-Binding Model

Semantic interpretation is evidence-bound when every claim in a generated narrative can be traced to a specific committed artifact, field, and value.

Evidence binding requires:
- Each signal claim cites: `dpsig_signal_set.json` → specific signal_id → specific signal_value
- Each cluster claim cites: `canonical_topology.json` → specific cluster_id → specific node_count
- Each domain claim cites: `semantic_topology_model.json` → specific domain_id → specific lineage/confidence
- Each readiness claim cites: `_classify_dpsig_readiness_state()` output → specific readiness_state
- Each zone claim cites: `binding_envelope.json` → specific zone_type

An LLM that generates a narrative without evidence binding is not operating in the semantic interpretation layer — it is hallucinating. The architecture must enforce evidence citation at the prompt-construction layer, not trust the LLM to self-govern.

### 5.5 Semantic Interpretation vs. Structural Truth Generation

This distinction is the most important boundary in the architecture:

| Structural Truth Generation (PROHIBITED for LLMs) | Semantic Interpretation (PERMITTED for LLMs) |
|---|---|
| "This cluster's CPI is 3.2" (inventing a value) | "This cluster's CPI of 3.2 suggests..." (citing a value) |
| "DOMAIN-05 is at high risk" (classification claim) | "DOMAIN-05's structural label shows elevated node density" (interpretation) |
| "This node is a pressure hotspot" (activation claim) | "The PSIG-001=2.32 signal indicates this node exhibits high structural outlier characteristics" |
| "The topology has changed" (mutation claim) | "Comparing the current and previous run snapshots, the cluster distribution has shifted" |
| "The system needs X" (prescription claim) | "Based on the CPI and CFA evidence, teams responsible for CLU-01 may wish to consider..." |

**Governing rule:** Interpretation is evidence-bound commentary. Truth generation is forbidden.

## 8. Agent Safety Boundaries

### 8.1 Permanent Safety Doctrine

The following safety constraints are permanent. They cannot be overridden by stream contracts, user requests, or product priorities. They exist to protect the deterministic sovereignty that makes this platform enterprise-grade.

**Rule AS-01 — Threshold Immutability**
No agent may suggest, request, compute, or apply modifications to any signal threshold (DPSIG: CPI_HIGH, CFA_DOMINANT; PSIG: THRESHOLD=2.0). Threshold modification requires SCRIPT_VERSION increment and a formal governance stream.

**Rule AS-02 — Signal Sovereignty**
No agent may generate, modify, or invalidate signal values. Signals are derived by deterministic scripts. An agent that contradicts a signal value is in violation — the signal value is correct; the agent's interpretation may be corrected.

**Rule AS-03 — Topology Immutability**
No agent may propose or execute changes to canonical_topology.json or binding_envelope.json. Topology is sovereign. Agents that identify topology concerns may surface them for human action through a governance stream.

**Rule AS-04 — Activation Authority Exclusion**
No agent may change an activation_state. Activation is owned by deterministic scripts. Agents that claim a node "should be" activated or deactivated are generating non-authoritative output and must be corrected.

**Rule AS-05 — Readiness Gate Exclusion**
No agent may override, bypass, or reclassify a readiness state. If an agent's reasoning suggests a DIAGNOSTIC_ONLY client should be treated as EXECUTIVE_READY, the agent is wrong. The gate is authoritative.

**Rule AS-06 — Evidence Suppression Prohibition**
No agent may omit, downweight, or selectively present structural evidence. Evidence-first discipline applies to all agent outputs. An agent that softens a HIGH-pressure finding without citing explicit structural justification is suppressing evidence.

**Rule AS-07 — Replay Identity Preservation**
No agent output may modify the TAXONOMY-01 fields: signal_value, activation_state, signal_stable_key, derivation_hash, derivation_trace. These fields are immutable once committed.

**Rule AS-08 — Semantic Authority Containment**
No agent may claim semantic authority that exceeds its grounding lineage. An agent reasoning about a NONE-lineage domain may only use structural labels. An agent cannot bootstrap semantic authority from adjacent EXACT-lineage domains.

**Rule AS-09 — Autonomous Remediation Prohibition**
No agent may execute remediation actions. Agents may suggest, analyze, and present options. Execution is always a human decision. There is no "apply remediation" action in this architecture.

**Rule AS-10 — Memory Governance**
Agent session memory is ephemeral unless explicitly captured as a governed artifact with full evidence lineage. An agent's memory of prior sessions does not constitute platform truth. Persistent agent memory requires a bounded memory governance contract.

## 9. Replay & Governance Requirements

### 9.1 Replay Governance Model

This platform operates two distinct types of replay:

**Type 1 — Structural Deterministic Replay (TAXONOMY-01)**
- Input: committed structural artifacts (canonical_topology.json, binding_envelope.json)
- Process: deterministic derivation scripts
- Output: bit-identical signal_value, activation_state, derivation_hash
- Guarantee: ANY replay on the same input produces IDENTICAL output
- LLM involvement: NONE — deterministic scripts only

**Type 2 — Semantic Evidence-Bound Replay**
- Input: committed structural artifacts + committed semantic_topology_model.json + committed prompt template
- Process: LLM reasoning over structured evidence
- Output: equivalent interpretation (not bit-identical prose, but same evidence citations, same confidence constraints)
- Guarantee: Same evidence + same prompt template → same evidence-binding scope; prose varies within bounds
- LLM involvement: BOUNDED — evidence injection mandatory; LLM reasons within grounding constraints

These two replay types must never be conflated. Type 1 is used for signal certification. Type 2 is used for narrative governance.

### 9.2 Semantic Lineage Model

Every semantically interpreted output must carry:

```json
{
  "interpretation_type": "semantic_narrative",
  "readiness_state_at_generation": "EXECUTIVE_READY",
  "evidence_sources": [
    {
      "artifact": "artifacts/dpsig/blueedge/run_blueedge_productized_01_fixed/dpsig_signal_set.json",
      "field": "signals[DPSIG-031].signal_value",
      "value": 2.1176
    }
  ],
  "grounding_constraints_applied": ["ALI-01", "ALI-02", "Q-00"],
  "confidence_floor": "STRONG",
  "generation_timestamp": "2026-05-08T...",
  "prompt_template_version": "narrative-v1.0"
}
```

This lineage record enables:
- Audit of what evidence was used
- Verification that confidence constraints were applied
- Reproduction of the interpretation scope (not the prose, but the constraint set)

### 9.3 Prompt Accountability Structure

Prompts that invoke LLM reasoning over structural evidence are governed artifacts, not free-form text.

Requirements for governed prompts:
1. Prompt template is committed to the repository with a version identifier
2. Evidence injection is standardized: `{signal_values}`, `{cluster_summary}`, `{readiness_state}`, `{grounding_constraints}`
3. Prompt explicitly includes the applicable aliasing rules and confidence constraints
4. Prompt explicitly prohibits fabrication and requires evidence citation
5. Prompt version is logged with every LLM output

Ungoverned prompts (ad-hoc, session-local, not committed) may be used for exploration but their outputs are not governed artifacts.

### 9.4 Orchestration Replay Contract

For multi-step agentic orchestration (agent chains, multi-turn investigation):

1. Each step must log: input artifact hash, operation type, output
2. The orchestration chain must be reproducible: same starting evidence → same investigation path
3. Orchestration logs are evidence artifacts if they inform executive outputs
4. An orchestration that modifies evidence is a governance breach (agents read; they do not write structural artifacts)
5. Orchestration memory that persists across sessions requires a bounded memory governance contract

### 9.5 Semantic vs. Structural Replayability — Explicit Distinction

| Property | Structural (L1–L2) | Semantic (L3–L4) |
|---|---|---|
| Bit-identity guarantee | YES — mandatory (TAXONOMY-01) | NO — prose varies |
| Evidence-identity guarantee | YES — mandatory | YES — mandatory (same evidence citations) |
| Tool to verify | replay_diff.json (overall_verdict: IDENTICAL) | evidence_lineage check (same source artifacts) |
| What breaks it | Any input file change | Prompt change OR evidence artifact change |
| LLM involvement | NONE | BOUNDED |
| Certification artifact | dpsig_signal_set.json + replay_diff.json | semantic_lineage.json + prompt_template.md |
