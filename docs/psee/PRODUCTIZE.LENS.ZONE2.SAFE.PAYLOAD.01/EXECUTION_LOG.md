## SECTION 6 — ENFORCEMENT ARCHITECTURE

`_validate_zone2_payload(payload)` runs as a defense-in-depth gate after Stage 5 output validation and before any payload is returned. It is a no-op for ZONE-1 and ZONE-3 payloads.

Six sequential checks:
1. Named forbidden fields absent
2. No forbidden key patterns (`_path`, `_file`, `_graph`)
3. Top-level keys ⊆ `ZONE2_L1_ALLOWED_FIELDS`
4. Signal subblock: `signal_id` absent; keys ⊆ `ZONE2_SIGNAL_ALLOWED_SUBFIELDS`
5. `value.raw` absent
6. `explanation` free of `ZONE2_FORBIDDEN_CONTENT_SUBSTRINGS`

Any check failure returns `{ error_type: "ZONE2_PROFILE_VIOLATION", reason: "ERROR_ZONE2_PROFILE_VIOLATION" }` — no payload is returned.

The Stage 2 zone filter remains the primary enforcement mechanism. `_validate_zone2_payload` is the formal boundary contract check and regression guard.
