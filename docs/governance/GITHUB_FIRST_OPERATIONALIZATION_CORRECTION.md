## §2. Why GitHub First

GitHub is the correct first external evidence operationalization target because it is substrate-native.

### Structural Signal Alignment

| GitHub Construct | PiOS/DPSIG Construct | Ontology Friction |
|-----------------|---------------------|-------------------|
| Repository topology | Architectural domain boundaries | LOW — near 1:1 |
| Dependency manifests | Coupling edges, propagation pathways | LOW — direct structural mapping |

| Import/dependency graphs | Structural debt, gravity concentration | LOW — direct computation |

### What GitHub Operationalization Proves

1. **External evidence portability** — the substrate processes evidence from a source other than advisory-curated archives
2. **Topology reconstruction portability** — structural domains are reconstructed from external evidence, not hardcoded to BlueEdge's 17-domain model
3. **Qualification portability** — Q-class and S-state operate on externally-sourced evidence without modification to qualification engines
4. **Marketplace universality** — the same substrate serves different evidence sources through L0 Ledger Selector modes
5. **Substrate independence from BlueEdge** — Program Intelligence is not a single-client artifact but a generalizable structural intelligence discipline

### Why GitHub Beats Jira as First Target

GitHub is a structural evidence environment. Jira is an operational representation layer.

| Dimension | GitHub | Jira |
|-----------|--------|------|
| Signal alignment | DPSIG near 1:1 | Requires heavy ontology bridging |
| Structural density | Repos, deps, owners, imports = topology | Epics, stories, sprints = work items |
| Data integrity | Machine-generated (commits, deps) | Human-declared (status, RAG) |
| Ontology friction | LOW — repo ≈ domain | HIGH — Epic ≠ Domain, Story ≠ Capability |
| Political contamination | NONE — code structure is structural truth | HIGH — statuses reflect what managers want to see |
| Automation viability | Structured API, machine-parseable | Semi-structured, requires human validation gate |

## §3. Why Jira Is Deferred

Jira is deferred to Phase 1A (PMO commercial entry) because it introduces complexity that does not belong in the first operationalization proof:

1. **Governance ontology ambiguity** — Jira models work (Epic → Story → Task). PiOS models structure (Domain → Capability → Component). Mapping is lossy and assumption-heavy.
2. **Work-item abstraction mismatch** — a Jira Story is not a structural capability. Structural domain reconstruction from work items requires an ontology bridge (Structural Domain Mapping Reference) that is itself a new governed artifact.
3. **Semantic inconsistency** — naming conventions, component usage, label taxonomy vary wildly across Jira instances. No two Jira projects are structurally identical.
4. **Representation-layer distortion** — Jira statuses reflect what humans declare, not what structurally exists. This introduces political contamination into evidence intake.
5. **Human validation gate required** — Jira evidence requires a Jira Evidence Qualification Gate (L0) and advisory curation before structural intelligence is meaningful. This is additional infrastructure that adds complexity to the first proof.

GitHub evidence is cleaner, denser, and structurally closer to the substrate. First operationalization should use the cleanest evidence source to isolate substrate portability questions from evidence quality questions.
